# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

`ENTSOE.jl` is a Julia client for the [ENTSO-E Transparency Platform RESTful API](https://transparencyplatform.zendesk.com/hc/en-us/sections/12783116987028-Web-API) (electricity market, generation, load, transmission, outage, balancing, master data for Europe). The package was scaffolded with [`OpenAPITemplate.jl`](https://github.com/langestefan/OpenAPITemplate.jl) (a PkgTemplates plugin around `openapi-generator`) on top of an OpenAPI 3.1 spec we derived from ENTSO-E's official Postman collection.

The user-facing surface is the *third* layer — named, code-prefilled wrappers in `src/conveniences/queries.jl` that return `Tables.jl`-compatible `StructVector`s. The codegen layer underneath is rarely the right thing to call directly; reach for it only when no named wrapper exists yet.

## Three layers, stacked

```text

    queries.jl    ←   ~45 named wrappers (day_ahead_prices, …, imbalance_prices,
                  ←     intraday_prices, year_ahead_forecast_margin,
                  ←     prices_of_activated_balancing_energy, SO-GL reserve family, …)

                  ←   `_query` is zip-aware — every wrapper transparently unzips
                      application/zip bodies (balancing 17.1.x, outages with many notices)
    parsing.jl    ←   XML → StructVector{NamedTuple} via EzXML
                  ←   parse_timeseries (price/quantity/*.amount), parse_timeseries_per_psr,
                      parse_installed_capacity, parse_unavailability (outages),
                      parse_master_data (registry), parse_acknowledgement, unzip_response

    codes.jl      ←   DOCUMENT_TYPE/PROCESS_TYPE/BUSINESS_TYPE/PSR_TYPE + code_for
    eic.jl        ←   EIC named tuple + EIC_REGISTRY + validate_eic
    period.jl     ←   entsoe_period(DateTime|Date|ZonedDateTime) -> Int64
    client.jl     ←   ENTSOEClient(token; validate_token=false) + entsoe_apis + the
                      pre_request_hook + is_uuid_token predicate
    config.jl     ←   ENTSOEConfig (package-wide defaults)
  client/         ← hand-written template overlay (also untouched by codegen)
    Client.jl     ←   the underlying Client type
    auth.jl       ←   NoAuth/BearerToken/APIKey/BasicAuth (we use NoAuth + the hook)
    errors.jl     ←   APIError hierarchy + check_response status → typed-exception map
    retry.jl      ←   RetryPolicy + with_retry  (default: 408/429/5xx, honours Retry-After)
    rate_limit.jl ←   TokenBucket + with_rate_limit

  api/            ← GENERATED by openapi-generator-cli — DO NOT EDIT
    ENTSOEAPI.jl  ←   inner module
    apis/         ←   one file per ENTSO-E group (Market, Load, …); 77 functions total
gen/
  regenerate.jl   ← rerun codegen against spec/openapi.json
spec/

  openapi.yaml    ← human-readable mirror
  Transparency Platform Restful API.postman_collection.json   ← original ENTSO-E source
scripts/
```

The generated `src/api/` directory carries `linguist-generated=true` in `.gitattributes`. Re-running `gen/regenerate.jl` nukes and re-creates `src/api/` only — nothing else.

### The one manual edit to a templated file

`src/ENTSOE.jl` is templated but `gen/regenerate.jl` does not touch it. We added one line at the end:

```julia
include("conveniences/conveniences.jl")
```

That umbrella include re-exports every name listed in `conveniences/conveniences.jl`'s `export` statements. If you ever re-run the *full* PkgTemplates scaffold (rare), you'll need to re-add this line.

## Postman → OpenAPI conversion

ENTSO-E publishes its API exclusively through a Postman collection — no official OpenAPI spec exists. The codegen pipeline needs OpenAPI, so we built a custom converter (`scripts/postman_to_openapi.jl`). Both the upstream source and the derived spec live in `spec/`:

- `spec/Transparency Platform Restful API.postman_collection.json` — the original Postman v2.1 export (~7.5 MB, 77 leaf requests across 8 groups). Treat it as the source of truth for parameter semantics: every parameter description starts with `[M]` (mandatory) or `[O]` (optional) and lists allowed enum values inline (e.g. `[M] A25 = Allocation results`).
- `spec/openapi.json` and `spec/openapi.yaml` — post-processed OpenAPI 3.1 spec. `gen/regenerate.jl` reads the JSON form directly.

### Why we didn't use a generic Postman→OpenAPI tool

We tried `postman2openapi` (the standard CLI). All 77 requests collapsed into a single path `/` with one giant `GET` operation, because OpenAPI keys operations by `path + method` and ENTSO-E uses the same URL for everything (only the query string differs). One method on the generated client — useless. Per-endpoint examples and `[M]`/`[O]` markers were also lost.

### What `scripts/postman_to_openapi.jl` does instead

1. **Walks the Postman folder tree** recursively; each leaf keeps the top-level folder (`Market`, `Load`, …) as its group.
2. **Synthesizes one path per request** of the form `/{group-slug}/{request-slug}` — purely so the OpenAPI key-uniqueness constraint is satisfied. `info.description` in the spec calls this out so downstream tools don't mistake the paths for real URLs.
3. **Translates per-parameter Postman metadata into OpenAPI fields.** `[M]` → `required: true`. The marker is stripped from the description but enum hints (`A25 = Allocation results`) are kept verbatim.
4. **Lifts `securityToken` out of every operation** into a `components.securitySchemes` entry (`apiKey, in: query`), applied at the root via `security`. This is what `ENTSOEClient`'s `pre_request_hook` keys off.
5. **Adds an `application/xml` 200 response** to every operation. Crucially, no schema is generated — which is why **no `APIModel` subtypes exist in `src/api/`** (`test/test-models.jl` records this with an `@info` rather than asserting). Parsing happens one layer up in `src/conveniences/parsing.jl`.

Dependency-self-bootstrapping: on first run the script activates `~/.julia/environments/entsoe-scripts` and adds `JSON3`, `YAML`, `OrderedCollections`. Output drops from postman2openapi's 10 MB YAML to ~180 KB.

### Re-running it

If ENTSO-E publishes an updated Postman collection, drop it in `spec/` with the same filename and run:

```bash
julia scripts/postman_to_openapi.jl     # rewrites spec/openapi.{json,yaml}
julia --project gen/regenerate.jl       # rebuilds src/api/ from spec/openapi.json
```

`gen/regenerate.jl`'s `SPEC_URL` is pinned to `spec/openapi.json`; `--from-file <path>` overrides it.

## How a typical query reads — prefer the named wrappers

```julia
using ENTSOE
using Dates

client = ENTSOEClient(ENV["ENTSOE_API_TOKEN"])

# Default: returns a Tables.jl-compatible StructVector
prices = day_ahead_prices(client, EIC.NL,
    DateTime("2024-09-01T22:00"),   # 2024-09-02 00:00 CET
    DateTime("2024-09-02T22:00"))

prices[1]            # → (time = DateTime("2024-09-01T22:00"), value = 41.27)
prices.value         # Vector{Float64}, no allocation
DataFrame(prices)    # works directly
```

The named wrappers (`day_ahead_prices`, `actual_total_load`, `day_ahead_load_forecast`, `installed_capacity_per_production_type`, `actual_generation_per_production_type`, `cross_border_physical_flows`, …) live in `src/conveniences/queries.jl`. They pre-fill the magic ENTSO-E codes (`A44`, `A65`, `A68`, …), accept `DateTime`/`Date`/`ZonedDateTime`/`Int64` for periods, and parse the XML response.

### `Parsed()` vs `Raw()` — keep returns type-stable

Every wrapper takes an optional trailing `ResponseFormat` positional argument:

```julia
prices = day_ahead_prices(client, EIC.NL, t1, t2)            # default: Parsed()
prices = day_ahead_prices(client, EIC.NL, t1, t2, Parsed())  # explicit
xml    = day_ahead_prices(client, EIC.NL, t1, t2, Raw())     # ::String
```

`Parsed` and `Raw` are subtypes of `abstract type ResponseFormat`, so each variant has a **concrete inferred return type** — no `Union` widening, no `Any`. Preserve this when adding new wrappers: give them a method on `Parsed` and a method on `Raw`, not a single method that branches on a flag.

### Long periods — use `query_split`

Most ENTSO-E endpoints reject single requests longer than one year (some cap at one day). To fetch multi-year history, wrap with `query_split`:

```julia
prices = query_split(day_ahead_prices, client, EIC.NL,
    DateTime("2020-01-01"), DateTime("2025-01-01");
    window = Year(1))     # use Day(1) for balancing-energy-bid endpoints
```

`query_split` calls the wrapper once per chunk, concatenates the results, and **catches `ENTSOEAcknowledgement` per chunk** so partially populated multi-year requests still return whatever data exists.

### When you must call the generated layer directly

For operations that don't yet have a named wrapper, drop down to the generated functions in `ENTSOE.ENTSOEAPI`:

```julia
apis = entsoe_apis(client)
xml, resp = ENTSOE.market121_d_energy_prices(
    apis.market, "A44", EIC.NL, EIC.NL, start_p, stop_p,
)   # returns (xml::String, ::ApiResponse); does NOT raise on Acknowledgement
```

Note: the generated layer returns `(xml, response)` tuples directly. It does **not** turn 200-with-Acknowledgement into an exception, and on non-2xx it returns `(nothing, resp)` instead of throwing. The named wrappers handle both via `_query_xml` (see `src/conveniences/queries.jl`).

## Key API conventions worth remembering

- **One real endpoint, many synthetic paths.** ENTSO-E's API is a single URL dispatched by query parameters. We split it into 77 OpenAPI operations under synthetic paths like `/market/12-1-d-energy-prices` purely so codegen produces one Julia function per logical query. The pre-request hook in `src/conveniences/client.jl` has a **second-stage** callback `(resource, body, headers)` that collapses the synthetic path back to `/api?<query>` before the request hits the wire. If you touch the spec or the hook, keep that rewrite working.
- **Auth is a query parameter, not a header.** `securityToken` is appended to the URL. The scaffold's built-in `APIKey` auth is header-only and does NOT work for ENTSO-E. `ENTSOEClient(token)` constructs a `Client` with `NoAuth()` and installs a `pre_request_hook` whose stage-1 form injects `ctx.query["securityToken"]` whenever an op declares the `"SecurityToken"` requirement.
- **Periods are `Int64` UTC `yyyyMMddHHmm`.** The generated layer accepts that alone. Named wrappers accept `DateTime`/`Date`/`ZonedDateTime`/`Integer` via `_to_period` → `entsoe_period`. ZonedDateTimes are converted to UTC first.
- **Bidding zones are EIC codes** (16-char). A curated 33-zone subset is exposed as `EIC.NL`, `EIC.DE_LU`, `EIC.NO2`, etc. The full registry is `EIC_REGISTRY` (mapping every EIC to `(name, types)` with types like `:BZN`, `:CTA`, `:MBA`). Pass `validate = true` to any wrapper to assert the zone exists and is the right type.
- **Code-list tables.** `DOCUMENT_TYPE`, `PROCESS_TYPE`, `BUSINESS_TYPE`, `PSR_TYPE` are NamedTuples of code→description. `code_for(TABLE, "wind onshore")` does reverse lookup by substring.
- **"No data" is HTTP 200.** ENTSO-E returns an `<Acknowledgement_MarketDocument>` body. The wrappers detect this via `check_acknowledgement` and re-raise as `ENTSOEAcknowledgement` (with `.reason_code` and `.text`). The generated layer does not; callers using it directly must run `check_acknowledgement(xml)` themselves.
- **`application/zip` happens.** Balancing 17.1.G/H/I, 1.2.3.F/H/I, and outages endpoints with many notices serve zipped XML. `_query` in `src/conveniences/queries.jl` sniffs the 4-byte ZIP magic, unzips with `ZipFile.jl` via `unzip_response`, runs the acknowledgement check per-member, and `vcat`-s the parsed `StructVector`s. `Raw()` returns the concatenated XML members separated by `<!-- next zip member -->` sentinels. This is fully transparent — no wrapper has to opt in.
- **HTTP errors are typed.** `src/client/errors.jl#check_response` maps status codes to `AuthError` (401/403), `RateLimitError` (408/429, parses `Retry-After`, extracts ENTSO-E's HTML ban message via `rate_limit_message`), `ClientError` (other 4xx), `ServerError` (5xx). Network failures become `NetworkError`; timeouts become `TimeoutError(:connect | :read | :total)`. All five have `Base.showerror` overloads that truncate HTML/long bodies — a 503-during-maintenance no longer dumps 100 KB into stack traces. Don't write `try/catch err isa Exception` blocks — match on the specific subtype.
- **Reliability middleware composes.** Use `with_defaults(...) do ... end` to wrap any call in retry → rate-limit → timeout, with logging on the outside. `RetryPolicy()` already retries on 408/429/5xx with `Retry-After` support, so for most cases you only need to add a `TokenBucket` and a `timeout`. See `src/client/middleware.jl`.

## Testing — BrokenRecord cassettes are everywhere

`test/cassettes/` holds ~100 YAML + a few BSON cassettes (BSON for endpoints that return `application/zip`, whose bytes don't roundtrip through YAML). Mode is decided **by file existence**, not env vars:

- Cassette missing → recording mode: real HTTP call, request + response saved to disk.
- Cassette present → playback mode: response replayed, request shape verified; no network.

To re-record, delete the cassette and run the tests with a valid token. Token resolution for recording: `ENV["ENTSOE_API_TOKEN"]` first, then `<repo-root>/token.txt` (single line, gitignored).

Shared BrokenRecord setup (loading via `Base.require`, padding `STATE` for Julia 1.12+'s `:interactive` thread pool, applying `configure!` with the standard `ignore_headers`/`ignore_query` redaction) lives in `test/_brokenrecord_helpers.jl`. Each consumer does:

```julia
include("_brokenrecord_helpers.jl")
const BR = _load_brokenrecord()
BR === nothing && (@info "BrokenRecord not installed — skipping"; return)
# `Base.require` advances the world; calls must go through `Base.invokelatest`.
```

### Two cassette naming conventions, same directory

- `load61_a_actual_total_load.yml` — function-named, exhaustively covering all 77 ops (driven by `test/_smoke_descriptors.jl` from `test-smoke.jl`). These prove the generated layer round-trips; content is **not** asserted.
- `load_61a_actual_total_load_NL.yml`, `tut_*.yml`, etc. — descriptive, used by `test-cassettes.jl` and `test-queries.jl` to assert on response content for representative endpoints.

To add a new endpoint to the smoke suite, see `scripts/regenerate_smoke_cassettes.jl`.

### Optional deps and the `OPENAPI_SKIP_LINTING` switch

`test/test-linting.jl` loads Aqua and JET, both of which are *not* in `test/Project.toml`. Each does a `Base.identify_package` probe and emits `@info "… not installed — skipping"` when absent. To install: `pkg> activate test` then `pkg> add Aqua@0.8 JET@0.11`. JET is gated to Julia 1.12.x.

`OPENAPI_SKIP_LINTING=1` makes `runtests.jl` skip the whole linting file (faster local loops).

### Cassette hygiene in pre-commit

YAML cassettes contain literal multiline tab-indented XML bodies that yamllint, yamlfmt, and the file-cleanup hooks would mangle — every file-touching hook in `.pre-commit-config.yaml` excludes `^test/cassettes/`. Don't loosen those excludes. Likewise `runic` (the Julia formatter) excludes `src/api/` to preserve byte-equality with codegen output.

## When in doubt

- Need a specific endpoint's parameter list and code values? Inspect the original Postman collection (`Transparency Platform Restful API.postman_collection.json`). The OpenAPI spec is derived from it but the Postman descriptions carry richer enum hints (`A25 = Allocation results` etc.) and the `[M]`/`[O]` markers.
- Need to know what `gen/regenerate.jl` will overwrite? Only `src/api/` — never `src/ENTSOE.jl`, `src/client/`, `src/conveniences/`, `test/`, or any of the docs.
- Adding a new named wrapper? Mirror an existing one in `src/conveniences/queries.jl` (e.g. `day_ahead_prices`): two methods (`Parsed`/`Raw`) for type stability, route through `_query_xml`, add the matching parser shape in `parsing.jl` if the document type isn't covered, export it in `conveniences.jl`, and add both a content-asserting test in `test-queries.jl` and a smoke descriptor in `test/_smoke_descriptors.jl`.
